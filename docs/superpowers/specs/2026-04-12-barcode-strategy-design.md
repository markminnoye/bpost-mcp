# Design: Customer Barcode Strategy Configuration

**Date:** 2026-04-12
**Status:** Approved
**Related issues:**
- markminnoye/bpost-mcp#16 — Customer onboarding workflow & barcode strategy configuration

---

## Goal

Allow tenants to configure how Mail ID barcodes are handled during submission. Currently `submit_ready_batch` defaults `genMID: "7"` (BPost generates 7-digit barcodes). This design adds tenant-level strategy configuration and MCP-generated barcode support.

---

## Barcode Strategies

| Strategy | `genMID` sent to BPost | `midNum` per Item | Requires |
|----------|----------------------|-------------------|----------|
| `bpost-generates` | `'7'` / `'9'` / `'11'` (from preference) | Not provided — BPost generates in response | Nothing extra |
| `customer-provides` | `'N'` | Required in CSV — customer maps a `midNum` column | Nothing extra |
| `mcp-generates` | `'N'` | Auto-generated by this service | `barcodeCustomerId` in credentials |

---

## Data Model

### 1. `bpost_credentials` — add column

| Column | Type | Nullable | Description |
|--------|------|----------|-------------|
| `barcode_customer_id` | `varchar(5)` | Yes | 5-digit BPost-assigned code for Mail ID program participation. Required for `mcp-generates` strategy. |

This is a credential (BPost-assigned identifier), not a preference — so it lives alongside `customer_number` and `account_id`.

### 2. New table: `tenant_preferences`

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | `uuid` | No | `gen_random_uuid()` | PK |
| `tenant_id` | `uuid` | No | — | FK → `tenants.id`, unique (one row per tenant) |
| `barcode_strategy` | `text` | No | `'bpost-generates'` | `'bpost-generates'` / `'customer-provides'` / `'mcp-generates'` |
| `barcode_length` | `text` | No | `'7'` | `'7'` / `'9'` / `'11'` — only applies to `bpost-generates` (controls `genMID` value) |
| `updated_at` | `timestamp` | No | `now()` | Last modification |

**Fallback for existing tenants:** No `tenant_preferences` row = `bpost-generates` with length `7`. Matches current behavior — zero breaking changes.

### 3. New table: `barcode_sequences`

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| `tenant_id` | `uuid` | No | — | FK → `tenants.id` |
| `week` | `integer` | No | — | ISO week number (1–53) |
| `next_value` | `integer` | No | `0` | Next batch sequence number |
| | | | | **UNIQUE(`tenant_id`, `week`)** |

One atomic increment per batch submission. Approximately 52 rows per tenant per year. Old rows can be cleaned up but are harmless if left.

### 4. Updated interfaces

**`BpostCredentials`** (in `get-credentials.ts`):
```ts
export interface BpostCredentials {
  bpostUsername: string
  bpostPassword: string
  customerNumber: string
  accountId: string
  prsNumber?: string
  barcodeCustomerId?: string  // NEW
}
```

**New `TenantPreferences`** (new file `get-preferences.ts`):
```ts
export interface TenantPreferences {
  barcodeStrategy: 'bpost-generates' | 'customer-provides' | 'mcp-generates'
  barcodeLength: '7' | '9' | '11'
}
```

**New `getTenantPreferences(tenantId)`**: Returns preferences or default `{ barcodeStrategy: 'bpost-generates', barcodeLength: '7' }` if no row exists.

## Barcode Generation (`mcp-generates` strategy)

### MID Number Structure

When strategy is `mcp-generates`, the service generates valid 18-digit MID numbers:

```
12  04521  15  042  000042
--  -----  --  ---  ------
FCC CustID WW  BBB  Counter
 2    5     2    3    6     = 18 digits
```

| Component | Digits | Source | Description |
|-----------|--------|--------|-------------|
| FCC | 2 | Fixed `12` | Customer-generated (`1`), 11-digit mail piece number (`2`) |
| BarcodeCustomerId | 5 | Tenant credentials | BPost-assigned 5-digit code, zero-padded |
| Week number (WW) | 2 | Submission timestamp | ISO week 01–53. Ensures 30-day uniqueness (weeks don't repeat within 52 weeks) |
| Batch sequence (BBB) | 3 | DB counter per (tenant, week) | 000–999. Atomic increment per batch submission |
| Row counter (NNNNNN) | 6 | Row index in batch | 000000–999999. Zero-padded position of the row |

### Capacity

- **1,000,000 rows per batch** (6-digit counter)
- **~143 batches per day** (1000 per week / 7)
- **30-day uniqueness guaranteed** by week number (doesn't repeat within 52 weeks)
- Human-readable: `12 04521 15 042 000042` = "Customer 04521, week 15, batch 42, row 42"

### Implementation: `src/lib/batch/generate-barcode.ts`

Pure function — no DB access, no side effects:

```ts
function generateMidNumber(
  barcodeCustomerId: string,  // 5 digits
  weekNumber: number,         // 1-53
  batchSequence: number,      // 0-999
  rowIndex: number,           // 0-999999
): string
```

Returns 18-digit string matching `MidNumberSchema` pattern (`/^[0-9]{14,18}$/`).

Validates:
- `barcodeCustomerId` is exactly 5 digits
- `weekNumber` is 1–53
- `batchSequence` is 0–999
- `rowIndex` is 0–999999

The DB sequence claim (getting `batchSequence`) happens in the caller, not in this function.

### Batch sequence claim

New function `claimBatchSequence(tenantId, weekNumber)`:

```sql
INSERT INTO barcode_sequences (tenant_id, week, next_value)
VALUES ($tenantId, $week, 1)
ON CONFLICT (tenant_id, week)
DO UPDATE SET next_value = barcode_sequences.next_value + 1
RETURNING next_value - 1 AS batch_sequence
```

Returns the batch sequence number (0-based). One DB write per batch submission.

## Wiring into `submit-batch.ts`

The strategy resolution happens in the `route.ts` handler, before calling `submitBatch()`. The service function itself does not change.

### Handler flow (additions to existing flow)

After loading batch state and credentials (existing steps 1–7), add:

1. **Load preferences:** `getTenantPreferences(tenantId)` — returns defaults if no row exists
2. **Resolve barcode strategy:**
   - **`bpost-generates`**: Set `genMID` to `preferences.barcodeLength`. Do not touch row `midNum`.
   - **`customer-provides`**: Set `genMID` to `'N'`. Validate every ready row has `midNum` in `row.mapped` — error if any are missing: `"Strategy is 'customer-provides' but N rows are missing midNum."`.
   - **`mcp-generates`**: Set `genMID` to `'N'`. Require `barcodeCustomerId` in credentials — error if missing: `"Strategy is 'mcp-generates' but barcodeCustomerId is not configured."`. Claim batch sequence from DB. Call `generateMidNumber()` for each row, inject `midNum` into `row.mapped`.
3. **Per-call override:** If the tool input explicitly provides `genMID`, it overrides the strategy-derived value. However, `midNum` injection still follows the strategy (providing `genMID: 'N'` manually does not trigger barcode generation — the tenant strategy controls that).

### `check-batch.ts`

No changes. MailingCheck is for address validation (OptiAddress), not barcode handling. It already hardcodes `genMID: 'N'`.

## Dashboard UI

### Credentials form: add `barcodeCustomerId`

Add to the existing "BPost-gegevens" form, after the PRS-nummer field:

- **Label:** "Barcode-klant-ID (optioneel)"
- **Help text:** "Een 5-cijferige code die je van bpost ontvangt als je deelneemt aan het Mail ID-programma. Alleen nodig als je barcodes automatisch wilt laten aanmaken door deze dienst."
- **Input:** `pattern="\d{5}"`, `maxLength={5}`, `title="Exact 5 cijfers"`
- **Validation in server action:** `/^\d{5}$/` or empty

### New section: "Barcode-instellingen"

New card below "BPost-gegevens":

**Title:** "Barcode-instellingen"
**Intro text:** "Kies hoe de barcodes voor je mailings worden aangemaakt."

**Radio group:**
1. "bpost maakt de barcodes aan" — default, selected for existing tenants
2. "Ik lever zelf barcodes aan in mijn adresbestand"
3. "Laat deze dienst de barcodes automatisch aanmaken" — shows note if `barcodeCustomerId` is not set: "Hiervoor heb je een Barcode-klant-ID nodig (zie hierboven)."

**Barcode length dropdown:** Only visible when strategy is `bpost-generates`. Options: "7 cijfers", "9 cijfers", "11 cijfers". Default: "7 cijfers".

**Server action:** Upsert into `tenant_preferences`.

## Not In Scope

- `check-batch.ts` — no barcode changes (address validation only)
- Onboarding wizard — separate UX effort
- Default format/priority/mode preferences — future columns on `tenant_preferences`
- `add_protocol_rule` for custom barcode generation rules — separate tool
- Barcode printing/rendering — physical print is customer's responsibility
